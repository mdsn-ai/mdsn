# SDK Error Fragments

This page lists the built-in failure prompt fragments generated by the SDK itself.

Scope:

- framework/host runtime behavior (`createHostedApp`)
- session/login guidance helpers from `@mdsnai/sdk/server`
- transport/protocol failures handled by SDK internals

Out of scope:

- app-specific business failures returned by your own action handlers

## 1. Contract Summary

- action handlers return Markdown fragments (including business failures)
- SDK host wraps framework/protocol failures into Markdown guidance fragments
- there is no protocol-level `errorCode` field in these SDK fragments

Use:

- HTTP status for machine-level handling
- Markdown body for agent/human guidance

## 2. Built-in SDK Failure Fragments

| Kind | HTTP status | Trigger | Default fragment body | Source |
| --- | --- | --- | --- | --- |
| `actionNotAvailable` | `404` | Page declares an action target, but the server has no handler for that action id | `## Action Status` + `This action is not available on the current server.` | `sdk/src/framework/hosted-app.ts` |
| `unsupportedContentType` | `415` | `POST` action request without `Content-Type: text/markdown` | `## Action Status` + `Unsupported content type for write action.` + resend guidance | `sdk/src/framework/hosted-app.ts` |
| `internalError` | `500` | Action throws unexpectedly (including invalid return type from handler) | `## Action Status` + `The action failed due to an internal error.` + raw error message | `sdk/src/framework/hosted-app.ts` |
| `authRequired` | `401` | App/session layer needs login guidance | `## Login Status` + `Login required: sign in before continuing.` | `sdk/src/server/error-fragments.ts` |

Notes:

- invalid action return type now surfaces as a `500` fragment with message: `Invalid action result: expected markdown string`
- this message is produced by `executeActionHandler()` and then wrapped by hosted-app

## 3. SDK Helpers

Available from `@mdsnai/sdk/server`:

- `renderErrorFragment()`
- `renderActionNotAvailableFragment()`
- `renderUnsupportedContentTypeFragment()`
- `renderInternalErrorFragment()`
- `renderAuthRequiredFragment()`

These helpers still return Markdown fragments only. There is no JSON error envelope.

## 4. App-Level Override

`createHostedApp()` / `createSiteApp()` / `createFrameworkApp()` now accept:

```ts
errorFragments: {
  actionNotAvailable?: (ctx) => string;
  unsupportedContentType?: (ctx) => string;
  internalError?: (ctx) => string;
}
```

Use this when you want app-specific wording while preserving the same HTTP status.

For session/login failures, keep returning your own auth fragment through `requireSessionFromCookie()`.

## 5. Non-Fragment Error Responses

These are errors, but not Markdown prompt fragments:

- `404 text/plain` `Not Found` for unmatched route paths (`createHostedApp` fallback)
- `406 text/plain` `Not Acceptable` when `Accept` is not supported for page rendering (`renderHostedPage`)

## 6. Is There an Error Code Mapping?

No protocol error-code mapping exists in SDK failure fragments now.

- no `errorCode` field in response body
- no `fieldErrors` object in response body
- rely on `HTTP status + Markdown guidance`
