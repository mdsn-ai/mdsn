# SDK Error Fragments

This page lists the built-in failure prompt fragments generated by the SDK itself.

Scope:

- framework/host runtime behavior (`createHostedApp`)
- transport/protocol failures handled by SDK internals

Out of scope:

- app-specific business failures returned by your own action handlers

## 1. Contract Summary

- action handlers return Markdown fragments (including business failures)
- SDK host wraps framework/protocol failures into Markdown guidance fragments
- there is no protocol-level `errorCode` field in these SDK fragments

Use:

- HTTP status for machine-level handling
- Markdown body for agent/human guidance

## 2. Built-in SDK Failure Fragments

| HTTP status | Trigger | Fragment body | Source |
| --- | --- | --- | --- |
| `404` | Page declares an action target, but the server has no handler for that action id | `## Action Status` + `This action is not available on the current server.` | `sdk/src/framework/hosted-app.ts` |
| `415` | `POST` action request without `Content-Type: text/markdown` | `## Action Status` + `Unsupported content type for write action.` + `Use Content-Type: text/markdown...` | `sdk/src/framework/hosted-app.ts` |
| `500` | Action throws unexpectedly (including invalid return type from handler) | `## Action Status` + `The action failed due to an internal error.` + raw error message | `sdk/src/framework/hosted-app.ts` |

Notes:

- invalid action return type now surfaces as a `500` fragment with message: `Invalid action result: expected markdown string`
- this message is produced by `executeActionHandler()` and then wrapped by hosted-app

## 3. Non-Fragment Error Responses

These are errors, but not Markdown prompt fragments:

- `404 text/plain` `Not Found` for unmatched route paths (`createHostedApp` fallback)
- `406 text/plain` `Not Acceptable` when `Accept` is not supported for page rendering (`renderHostedPage`)

## 4. Is There an Error Code Mapping?

No protocol error-code mapping exists in SDK failure fragments now.

- no `errorCode` field in response body
- no `fieldErrors` object in response body
- rely on `HTTP status + Markdown guidance`
